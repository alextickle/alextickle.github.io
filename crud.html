Design

This application was an assignment for UCSD Extension's Java Programming III class (Intermediate Java Programming, CSE-40481). The goal of the assignment was to build a CRUD (Create, Retrieve, Update, Delete) inventory management program for a 'Borders'-style retailer using Model View Controller (MVC) architecture.

The view contains logic for handling user input and displaying data received from the model. Its implementation has been hidden from the model and controller components so that it could readily be swapped out for a different view as long as the new view implements the Viewable interface. The initial view was text-based and all input was taken from the command line. I built a GUI view for the second part of the assignment using the Swing library.

The MediaItem class defines fields and methods that are used by its child classes (CD, DVD, and Book). The components communicate with each other by sending and editing Command objects, which are basically just MediaItems wrapped in objects so that can include info on the type of user request. The view handles user input and creates and appropriate Command object to pass to the controller, which in turn passes it to the Model. The model uses the Command object to update itself and then alerts the view that the update is complete. Then the view responds with a more specific request for data, which the model then exports to the view for display to the user.

The initial model stored all data in an ArrayList of Properties objects. It reads and writes data to a local text file (inventory.txt). If there is no data file in the current directory at runtime then the application creates a new one. The gitHub repository and included .jar file contain an inventory.txt file that is prepopulated with data so that the applicationâ€™s functionality can be more readily demonstrated. The inventory file is saved after each update to the model. A second model that makes use of a database via JDBC was built for the third step of the assignment

Challenges

Working on this project clearly instilled upon me the advantage of developing a solid high-level plan for how the program will work before writing a single line of code. Luckily this advantage became apparent fairly early on in the process, and once I had a clear plan implementing the individual component methods became much simpler. Before writing this program I had minimal experience with the filesystem libraries and so it took a lot of experimenting to develop a working persistance layer. The GUI turned out to be pretty straightforward, although all of the logic of the text-based view had to be completely rewritten to accomodate event-handling.
